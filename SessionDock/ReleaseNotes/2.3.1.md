SessionDock 2.3.1

More reliable settings and account cleanup

- Settings changes now save in order without blocking the window. Rapid destination edits are combined safely, while the final edit is still saved when focus changes and included in SessionDock's bounded close-time save.
- If a settings write is interrupted, SessionDock can recover from the last valid local backup. When possible, unreadable files are preserved for diagnosis, and uncertain settings never trigger automatic browser-profile deletion.
- Removing an account now records the cleanup before changing the account list. Interrupted WebView2 profile deletion is retried on the next launch, and unsafe or out-of-folder paths are rejected.
- Startup now uses a two-second budget when retrying previously interrupted profile cleanup. Locked profiles remain recorded for another attempt instead of accumulating minutes of retry delays. Cleanup of incomplete sign-in profiles uses the same bounded, cancelable behavior.
- Imported startup sounds are staged until the matching settings change is safely stored. Recovery protects sounds referenced by either the current settings or the backup.

More predictable failure and shutdown behavior

- Expected disk, permission, WebView2, and browser-process failures in interactive account, settings, launch, and browser actions are now contained and reported instead of escaping through a window action.
- Closing SessionDock cancels active work and waits up to two seconds for asynchronous operations and best-effort settings and profile cleanup, reducing delays from stalled background work.
- An interrupted application-data migration now leaves a durable guard that pauses automatic browser-profile cleanup until the data roots are reconciled.
- Acknowledged recovery warnings no longer reopen on every launch while their cleanup protection remains active; they return if the protected state clears and later recurs.
- HandleScope startup and connection checks now track the verified per-user API process and reject duplicate, stale, or unverified starts.
- Roblox Player and HandleScope child processes no longer inherit optional launch-hook endpoint or bearer-token environment settings.
- Release notes are now displayed as readable, inert text instead of raw Markdown while preserving selection, copying, keyboard scrolling, and screen-reader labeling.
- Update-file locks, network timeouts, permission failures, and rejected downloads now have specific recovery guidance instead of one generic error; unexpected internal faults remain observable.
- The installer now uses a separate SessionDockApp package identity, so application files no longer share either the current SessionDock data directory or the historic RobloxOne data directory.
- Upgrades from Roblox One or SessionDock 2.3.0 and earlier now use a side-by-side corrective install. SessionDock copies only recognized user data, preserves the entire legacy source as a fallback, never copies or moves installer files, and pauses orphan-profile cleanup until recovered accounts and sign-ins have been confirmed.
- The update guide no longer recommends the unsafe 2.1.5 Setup-over-existing bridge. It explains why older apps report an untrusted redirect and gives the data-preserving corrective path.

This patch adds no telemetry, new network destination, or bundled integration.
